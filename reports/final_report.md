## 1. Architecture summary

`ReliabilityGateway.complete()` routes every request through four layers, in
strict cheapest/safest-first order. Each layer can independently succeed,
fail, or be skipped, and the layer below only sees traffic that the layer
above could not satisfy.

```
User Request
    |
    v
[Gateway.complete(prompt)]
    |
    v
[Cache.get(prompt)] --------------------------> HIT (score >= threshold,
    | MISS / privacy-blocked / false-hit          not a false-hit)
    v                                              |
[CircuitBreaker: primary] --.                      |
    | allow_request()?      |                      |
    | OPEN -> skip           `-> provider.complete()--(success)--> [Cache.set()]
    | CLOSED/HALF_OPEN -> call                                          |
    v (ProviderError / CircuitOpenError)                                v
[CircuitBreaker: backup] --.                                    return GatewayResponse
    | same allow/call logic |                                   route="primary"|"cache_hit:x.xx"
    v (fails too)           `-> provider.complete()--(success)--> [Cache.set()]
    |                                                                   |
    v                                                                   v
[Static fallback message]                                       return GatewayResponse
route="static_fallback"                                          route="fallback"
error=last_error
```

- **Cache** (`ResponseCache` / `SharedRedisCache`) — n-gram cosine similarity
  lookup with privacy and false-hit guardrails. Answers instantly, no cost.
- **Circuit breaker** (`CircuitBreaker`, one per provider) — 3-state machine
  (CLOSED → OPEN → HALF_OPEN → CLOSED) that fails fast instead of hammering a
  broken provider.
- **Provider chain** (`FakeLLMProvider` primary → backup) — tried in order;
  the gateway moves to the next provider on `ProviderError` or
  `CircuitOpenError`.
- **Static fallback** — degraded-but-always-answers message when every
  provider is unavailable.

## 2. Configuration

| Setting | Value | Reason |
|---|---:|---|
| failure_threshold | 3 | Primary's baseline `fail_rate` is 0.25, so 3 *consecutive* failures is a low-probability event (~1.6%) under normal noise — high enough to avoid opening on a single blip, low enough to react to a real outage quickly. |
| reset_timeout_seconds | 2 | Compressed for the lab's simulated timescale (requests take ~0.2-0.3s); in a real deployment this would be tens of seconds, but 2s here lets a 100-request chaos run actually observe OPEN→HALF_OPEN→CLOSED recovery cycles within a few seconds. |
| success_threshold | 1 | One successful probe is enough evidence of recovery for this workload (idempotent read-only LLM completions); a higher value would be justified for a provider with expensive side effects, at the cost of slower recovery. |
| cache TTL | 300s | FAQ/policy-style answers (see `data/sample_queries.jsonl`) don't change minute-to-minute; 5 minutes captures repeat queries within a session/burst without serving stale answers for long. |
| similarity_threshold | 0.92 | Verified empirically (see below) that this threshold alone is **not** sufficient to reject all wrong matches — it is deliberately paired with `_looks_like_false_hit()` as a second guardrail rather than relied on in isolation. |
| load_test requests | 100 | Large enough per scenario to get a stable P95/P99 estimate, small enough that a 3-scenario chaos run finishes in under a minute. |

**Similarity threshold evidence** (computed directly against this repo's
`ResponseCache.similarity()`):

```
similarity("Summarize refund policy for 2024 deadline",
           "Summarize refund policy for 2026 deadline") = 0.9375
```

This pair differs only by year but scores **above** the 0.92 threshold —
proving that no similarity-threshold value can, by itself, distinguish
"same words, different year" from "genuinely the same question." This is
exactly why `_looks_like_false_hit()` exists as a mandatory second layer
(see `test_false_hit_detection_different_years`), not as a nice-to-have.

## 5. Cache comparison

Same `configs/default.yaml` (all 3 scenarios, 300 requests total), run once
with `cache.enabled: true` and once with `cache.enabled: false`:

| Metric | Without cache | With cache | Delta |
|---|---:|---:|---|
| latency_p50_ms | 274.14 | 270.45 | -3.7 ms |
| latency_p95_ms | 316.96 | 313.35 | -3.6 ms |
| estimated_cost | 0.124142 | 0.045402 | **-63.4%** |
| cache_hit_rate | 0.0 | 0.6267 | +0.6267 |
| circuit_open_count | 21 | 7 | -14 |

**Analysis:**
- The dominant benefit of caching here is **cost**, not the P50/P95 numbers
  shown above — those percentiles only include requests that actually hit a
  provider (`latencies_ms` only appends when `latency_ms > 0`, so cache hits,
  which return in 0ms, are excluded from the percentile array). The real
  end-user latency win is much larger than -3.7ms once you account for 62.7%
  of requests returning instantly; the current metric just doesn't surface
  it directly — a gap worth fixing (see Failure analysis / Next steps).
- Circuit opens dropped from 21 to 7 with cache enabled. This isn't a
  circuit-breaker behavior change — it's a side effect of caching removing
  62.7% of requests from ever reaching a provider, so there are simply fewer
  chances for the (probabilistic) `FakeLLMProvider` failures to accumulate
  toward `failure_threshold`.
- The `estimated_cost_saved` field in `metrics.json` (0.189) is a flat
  per-cache-hit estimate (`+= 0.001` per hit) and does not exactly match the
  real cost delta measured here (0.124142 - 0.045402 = 0.0787). The flat
  estimate overstates savings for this config because it doesn't account for
  each provider's actual `cost_per_1k_tokens`; the two-run comparison above
  is the more trustworthy number.

## 6. Redis shared cache

- **Why in-memory cache is insufficient for multi-instance deployments:**
  `ResponseCache._entries` is a plain Python list held in one process's
  memory. If the gateway runs as multiple worker processes/instances behind
  a load balancer, each instance builds its own cache from scratch — the
  same query hitting two different instances is two cache misses, and one
  instance can't benefit from another's cached answers.
- **How `SharedRedisCache` solves this:** cache entries are stored as Redis
  hashes (`{prefix}{md5(query)}` → `{query, response}`) with `EXPIRE` for
  TTL, so every gateway instance reading/writing the same Redis instance
  shares one cache. No in-process eviction loop is needed — Redis expires
  keys automatically.

### Evidence of shared state

Two independent `SharedRedisCache` instances, same Redis URL and prefix,
run in the same process to simulate two separate gateway workers:

```
c1 = SharedRedisCache(redis_url, prefix="rl:cache:report:")
c2 = SharedRedisCache(redis_url, prefix="rl:cache:report:")   # separate instance
c1.set("Explain circuit breaker states in one paragraph.", "A circuit breaker has three states...")
cached, score = c2.get("Explain circuit breaker states in one paragraph.")
# => cached='A circuit breaker has three states...' score=1.0
```

`c2` never called `set()` — it only ever read from Redis — yet it returned
`c1`'s cached value with a perfect exact-match score, confirming the two
instances share state through Redis rather than through process memory.

## 8. Failure analysis

**Weakness: the circuit breaker is not safe under concurrent access, and
HALF_OPEN does not limit concurrent probes to one.**

`CircuitBreaker.allow_request()` returns `True` for *every* caller while
`state == HALF_OPEN` — there's no single-flight guard (lock, token, or
atomic compare-and-swap) restricting the probe to one in-flight request.
Under the lab's sequential `run_scenario()` loop this is invisible because
only one request is ever in flight at a time. But the moment concurrency is
introduced (the README's own stretch goal: `ThreadPoolExecutor` in
`run_simulation`), multiple threads could all read `state == HALF_OPEN`
simultaneously and all be allowed through — defeating the entire point of
HALF_OPEN, which is to test recovery with a *single* low-risk probe before
trusting the provider with full traffic again. Worse, `record_success()` and
`record_failure()` mutate `failure_count`/`success_count`/`state` without
any lock, so concurrent successes and failures racing on the same
`CircuitBreaker` instance could interleave in ways that produce an
inconsistent state transition (e.g., a failure recorded after the state has
already flipped to CLOSED from a concurrent success).

**Proposed fix:** add a `threading.Lock` around the state-mutating sections
of `allow_request()`/`record_success()`/`record_failure()`, and change
HALF_OPEN admission to a compare-and-swap that only lets the *first* caller
through per probe window (subsequent concurrent callers get treated as if
OPEN until the probe resolves). This should be validated with a
`ThreadPoolExecutor`-based load test that asserts at most one HALF_OPEN
probe is in flight at a time — currently no test in this repo exercises the
breaker under concurrency, so this bug would not be caught by `make test`
today.

## 9. Next steps

1. Add thread-safety (lock + single-flight HALF_OPEN admission) to
   `CircuitBreaker`, plus a concurrent load test (`ThreadPoolExecutor`) that
   would have caught the race described above — currently the only load
   testing is the sequential loop in `run_scenario()`.
2. Fix the latency metric blind spot from §5: report an "all-requests"
   average/percentile that includes 0ms cache hits (in addition to the
   current provider-only percentile), so the report reflects the true
   end-user-facing latency benefit of caching, not just provider latency.
3. Implement cost-aware routing (mentioned as a bonus TODO in
   `gateway.py`): track cumulative `estimated_cost` and, once a configured
   budget threshold is crossed, skip the more expensive primary provider in
   favor of cache-only or the cheaper backup — directly useful given §5
   shows cache disabled nearly triples cost.
